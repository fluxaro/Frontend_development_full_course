DNS — The Phone Book of the Internet

DNS stands for Domain Name System. It translates human-readable domain names into IP addresses that computers use to find each other.

The Problem DNS Solves

Computers communicate using IP addresses like 140.82.121.4. Humans remember names like github.com. DNS bridges that gap.

Without DNS, you would have to type 140.82.121.4 instead of github.com.

How DNS Resolution Works

  1. You type github.com in your browser

    The browser needs to find the IP address for this domain.

  2. Browser checks its own cache

    Has it looked up this domain recently? If yes, use the saved answer.

  3. Operating system checks its cache

    The OS keeps its own DNS cache too.

  4. Query sent to Recursive Resolver

    Your ISP's DNS server receives the query and starts searching.

  5. Root Name Server consulted

    The resolver asks a root server: "Who handles .com domains?" Root server replies with the address of the .com TLD server.

  6. TLD Name Server consulted

    The resolver asks the .com server: "Who handles github.com?" It replies with GitHub's authoritative name server.

  7. Authoritative Name Server answers

    GitHub's own DNS server replies: "github.com is at 140.82.121.4"

  8. IP address returned to browser

    The browser now knows where to connect. The answer is cached for next time.

Types of DNS Records

Common DNS Record Types
Record Type Full Name What It Does Example
A Address Record Maps a domain to an IPv4 address github.com → 140.82.121.4
AAAA IPv6 Address Record Maps a domain to an IPv6 address github.com → 2606:50c0::68
CNAME Canonical Name Alias — points one domain to another domain www.github.com → github.com
MX Mail Exchange Tells email servers where to deliver mail github.com → aspmx.l.google.com
TXT Text Record Stores text data, used for verification and security v=spf1 include:_spf.google.com ~all
NS Name Server Specifies which servers are authoritative for this domain ns1.github.com

DNS Caching

DNS answers are cached (saved temporarily) to speed things up. Each record has a TTL (Time To Live) value — the number of seconds the answer can be cached before it must be looked up again.

  • Short TTL (e.g. 300 seconds) — changes take effect quickly
  • Long TTL (e.g. 86400 seconds = 1 day) — faster lookups, slower updates

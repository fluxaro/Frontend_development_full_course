# Class Work — Exploring a Real GitHub Repository

## Goal

You're going to visit a real, live open source project on GitHub and explore it like a developer would. This will make the concepts from the README feel real and concrete.

**Time:** 20–30 minutes  
**Works best:** In pairs or small groups

---

## The Repo You'll Explore

Go to: **https://github.com/facebook/react**

This is the actual source code for React — one of the most popular JavaScript libraries in the world. Millions of developers use it. You're about to look at the same code they do.

---

## Step-by-Step Exploration

### Step 1 — The Main Page

When you land on the repo, take a look around. Answer these questions:

1. What is the name of the repository? (format: owner/repo-name)
2. How many **Stars** does it have? (Stars = developers who bookmarked it)
3. How many **Forks** does it have? (Forks = developers who made their own copy)
4. What programming language is listed as the primary language?
5. Read the first paragraph of the README. What does it say React is?

---

### Step 2 — Commits

Click on the **commits** link (usually shows a number like "15,000+ commits").

1. How many total commits are there?
2. Look at the 3 most recent commits. What do their messages say?
3. Click on one commit. What do you see? (Look for green/red lines — those are additions/deletions)
4. Who made the most recent commit? What is their GitHub username?

**Write down:** What does a commit message tell you about what changed?

---

### Step 3 — Branches

Click on the **branches** dropdown (it probably says "main" by default).

1. How many branches are there?
2. What is the name of the default/main branch?
3. Can you find any branches with names that hint at what feature they're working on?

**Write down:** Why do you think a big project like React would have multiple branches?

---

### Step 4 — Issues

Click on the **Issues** tab.

1. How many open issues are there?
2. How many closed issues are there?
3. Look at 2–3 open issues. What kinds of problems are people reporting?
4. Do issues have labels? What labels do you see? (e.g., "bug", "enhancement", "good first issue")

**Write down:** What is an Issue used for in a GitHub project?

---

### Step 5 — Pull Requests

Click on the **Pull Requests** tab.

1. How many open pull requests are there?
2. Click on one pull request. What is it trying to change or add?
3. Can you see a conversation happening in the comments?
4. Is there a "Files changed" tab? What does it show?

**Write down:** What is a Pull Request? How is it different from just pushing code directly?

---

### Step 6 — The README

Scroll back to the main page and read the README more carefully.

1. Does the README have a table of contents or sections?
2. Does it have installation instructions?
3. Does it have links to documentation?
4. Does it have badges (little icons showing build status, version, etc.)?

**Write down:** Why is a good README important for an open source project?

---

## Bonus Exploration

If you finish early, explore one of these:

- **Actions tab** — Automated tests and workflows that run when code is pushed
- **Insights tab** — Statistics about contributors and activity
- **Code tab** — Browse the actual source files. Can you find any `.js` files?
- Find a **"good first issue"** label — these are issues specifically marked as beginner-friendly

---

## Discussion Questions

Share your findings with the class:

1. Were you surprised by how many commits/contributors this project has?
2. What did you learn from reading the commit messages?
3. How does GitHub make it possible for hundreds of developers to work on the same project?
4. Would you feel comfortable contributing to an open source project someday? What would you need to learn first?

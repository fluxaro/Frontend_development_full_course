Why Semantic HTML Is the Foundation of Good Web Development

By Sarah Chen · January 15, 2025 · 8 min read

Introduction

When developers first learn HTML, they often reach for <div> for everything. It works — the page renders. But it misses the point of HTML entirely. HTML is not just about making things appear on screen. It is about communicating meaning.

Semantic HTML means choosing elements based on what the content means, not how it looks. A <nav> is for navigation. An <article> is for a self-contained piece of content. A <button> is for actions.

The Problem with Div Soup

Here is what a non-semantic page looks like: 

<div class="header">
  <div class="nav">
    <div class="nav-item"><a href="/">Home</a></div>
  </div>
</div>
<div class="main">
  <div class="article">
    <div class="title">My Post</div>
  </div>
</div>

A screen reader sees a wall of divs with no structure. A search engine cannot tell what is important. A developer reading this code has to guess what each div does.

The Semantic Solution

Here is the same page written semantically:

<header>
  <nav>
    <ul>
      <li><a href="/">Home</a></li>
    </ul>
  </nav>
</header>
<main>
  <article>
    <h1>My Post</h1>
  </article>
</main>

Now a screen reader can announce "navigation landmark" and "main content area." A search engine knows the <h1> is the article title. A developer reading this code immediately understands the structure.

Key Semantic Elements

<header>
Introductory content. Can be used for the page header or an article header.
<nav>
Navigation links. Use for main navigation, breadcrumbs, and pagination.
<main>
The main content of the page. Only one per page.
<article>
Self-contained content that makes sense on its own — blog posts, news articles, comments.
<section>
A thematic grouping of content. Should have a heading.
<aside>
Content tangentially related to the main content — sidebars, pull quotes, ads.
<footer>
Footer for the page or a section. Contains copyright, links, author info.
